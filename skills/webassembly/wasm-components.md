# WASM Component Model

## Module vs Component — the Core Distinction

A **module** is the base unit of WebAssembly: a binary `.wasm` file containing code, memory, tables, and imports/exports. Imports and exports use core WASM types only: `i32`, `i64`, `f32`, `f64`, `funcref`, `externref`. There is no native concept of a string, a list, a record, or an error type at the module boundary. Passing a string means passing a pointer (i32) and a length (i32), with a convention on who allocated what and where.

A **component** is a higher-level wrapper around one or more modules. It describes its imports and exports using WIT types (strings, lists, records, variants, options, results, resources) via the component model ABI. The runtime — not the programmer — handles the serialization and memory layout for crossing the boundary.

Concretely:
- A raw module exports: `(func "drive" (param i32 i32 i32) (result f64))` — caller must know the ABI convention
- A component exports: `drive: func(ctx: context) -> signal` — the type is the ABI

In a typical embedding: the host crate defines a WIT world that describes what guest components export, then uses `wasmtime::component::bindgen!` to generate the Rust glue for invoking those exports.

## Binary Format of Components

Components use the same `.wasm` file extension but have a different binary magic prefix than modules:

- Module: `\0asm` (0x00 0x61 0x73 0x6D) followed by version 0x01 0x00 0x00 0x00
- Component: `\0asm` followed by version 0x0A 0x00 0x01 0x00

The structure inside a component binary is a tree of nested sections. A component can embed multiple modules, alias their exports, lift/lower types through the canonical ABI, and describe its interface in terms of WIT worlds and interfaces.

To inspect:
```
wasm-tools component wit some-component.wasm   # dump the WIT interface
wasm-tools validate some-component.wasm        # validate the binary
wasm-tools print some-component.wasm           # dump WAT (component text format)
```

## Interface Types vs Core WASM Types

The component model introduces **interface types** — a type system that lives above the core WASM type system. These types exist only at component boundaries; inside the module, all types are still i32/i64/f32/f64.

| Interface Type | Description | Core representation |
|----------------|-------------|---------------------|
| `bool` | Boolean | i32 (0 or 1) |
| `u8`/`u16`/`u32`/`u64` | Unsigned integers | i32 or i64 |
| `s8`/`s16`/`s32`/`s64` | Signed integers | i32 or i64 |
| `f32`/`f64` | Floats | f32 or f64 |
| `string` | UTF-8 string | (ptr: i32, len: i32) — copied |
| `list<T>` | Variable-length sequence | (ptr: i32, len: i32) — copied |
| `tuple<A, B, ...>` | Fixed-length heterogeneous | flattened or via memory |
| `record { ... }` | Named struct | flattened or via memory |
| `variant { ... }` | Tagged union | (discriminant: i32, payload) |
| `option<T>` | Optional value | (some: i32, value) |
| `result<T, E>` | Success or error | (ok: i32, value/error) |
| `resource` | Opaque handle | i32 handle index |

The **canonical ABI** defines exactly how each interface type maps to core WASM values at the crossing point. This mapping is generated by tooling (`wit-bindgen`, `wasmtime::component::bindgen!`) — never write it by hand.

## Canonical ABI Lifting and Lowering

When a host calls a component function with a `string` argument:
1. The host serializes the string into a buffer in the component's linear memory (via `canonical_abi_realloc` or the component's allocator).
2. The host calls the component function with (ptr, len) pointing into that buffer.
3. Inside the component, the runtime reconstructs a Rust `String` (or equivalent) from (ptr, len).

This is "lowering" (host → component). The reverse — returning a string from the component — is "lifting" (component → host). Both directions involve a memory copy.

**Performance implication:** every cross-boundary call with a `record` carrying lists copies the list payload. For a context struct with `list<measurement>` of ~100 entries this is negligible; for microsecond hot paths the copy cost dominates. Either reduce crossing frequency (batch) or move the hot loop into the host.

## Resources

Resources are opaque handles to objects that live on one side of the boundary (host or guest). Unlike records — which are value types copied across the boundary — resources are reference-counted handles. The runtime manages their lifecycle.

A resource handle looks like an i32 on the wire, but the component model enforces ownership: a handle can only be used by the component/host that owns it, and must be dropped explicitly.

Resources are appropriate when:
- The object is expensive to copy (e.g., a large ML model loaded in the host).
- The object has an inherent identity that must be preserved across calls.
- Mutable state on one side needs to be accessed by the other side without copying.

When in doubt, prefer records: pure-compute components that own all of their state in linear memory rarely need resource handles. Reach for resources when the host genuinely needs to hand the guest a long-lived object the host continues to own.

## Composition with `wac`

Composition connects one component's exports to another component's imports, producing a new composed component. No host code is needed for wired dependencies.

Tool: `wac` (WebAssembly Composition tool, from Bytecode Alliance)

```
# Plug component B's exports into component A's imports
wac plug component-a.wasm --plug component-b.wasm -o composed.wasm
```

The composition is declared in a `.wac` file (WAC is its own small language):
```
package composed:example;

let a = new example:component-a { "example:dependency/api": b.api };
let b = new example:component-b { };

export a...;
```

Composition is powerful but add-on. In simple architectures (one component per role, loaded directly by the host), composition is not used. Composition becomes relevant when multiple components need to be wired together before the host sees them — e.g., a logging adapter plugged into every domain component.

## `wasm-tools` Reference

`wasm-tools` is the Swiss Army knife for WASM binary manipulation. Install:
```
cargo install wasm-tools
```

Key subcommands:

| Command | Purpose |
|---------|---------|
| `wasm-tools validate` | Validate a module or component binary |
| `wasm-tools print` | Disassemble to text format (.wat) |
| `wasm-tools component wit` | Extract WIT interface from a component |
| `wasm-tools component new` | Wrap a module into a component (with adapter) |
| `wasm-tools compose` | Compose components (alternative to `wac`) |
| `wasm-tools strip` | Remove custom sections (debug info, names) |
| `wasm-tools demangle` | Demangle Rust symbol names in output |
| `wasm-tools metadata show` | Show producers and other metadata |

Typical inspection workflow:
```
# Did the build produce a component (not a raw module)?
wasm-tools validate --features component-model path/to/my_component.wasm

# What does the component export?
wasm-tools component wit path/to/my_component.wasm
```

## Host-Side Instantiation Pattern (wasmtime)

```rust
// This is the pattern — for Rust specifics, see the rust-wasm skill.
use wasmtime::component::{Component, Linker};
use wasmtime::{Engine, Store};

// 1. Create engine (reuse across instantiations)
let engine = Engine::default();

// 2. Compile component bytecode (expensive — cache this)
let component = Component::from_file(&engine, "my_component.wasm")?;

// 3. Build linker with required imports
let mut linker: Linker<StoreData> = Linker::new(&engine);
// Add WASI if the component requires it, or skip for pure-compute components

// 4. Create store (per-instance state)
let mut store = Store::new(&engine, StoreData {});

// 5. Instantiate (links imports, calls start function)
let bindings = GeneratedBindings::instantiate(&mut store, &component, &linker)?;

// 6. Call exported function (the exact path comes from your WIT package + interface name)
let result = bindings.my_org_my_pkg_plugin().call_invoke(&mut store, &ctx)?;
```

The code-generation step (`bindgen!`) is Rust-specific — see the rust-wasm skill for `wasmtime::component::bindgen!` details.

## When to Use Components vs Raw Modules

| Scenario | Component | Raw module |
|----------|-----------|------------|
| Interface uses strings, records, results | Yes | Painful manual ABI |
| Multiple implementations behind same WIT | Yes | No standard way |
| Composition required | Yes | No |
| Pure numeric interface, performance critical | Maybe | Yes |
| Browser (wasm-bindgen) | No | Yes |
| Hot-swappable plugins with structured input/output | Yes | No |

**Default to components for any modern Rust-to-WASM scenario** where the boundary carries structured data (records, lists, results). Raw modules would require manual memory layout negotiation for every field — components handle that for you via the canonical ABI.
